Week 4 Progress Report

Date: 9/11-9/17/2023

Total Hours: 13

Description of Design Efforts:

This weeks primary focus was defining the interfacing between each of the different subsystems. More specifically, deciding exactly how the microcontroller will send out/receive data and which IO pins must be reserved to accomplish this.

PC to Microcontroller (USB to UART) (4 hours)

Per Abby’s request, the PC would hopefully use the easily accessible USB ports on the side of whatever computer we end up going with (PowerPC? Powerful PC? Much debate on what we should call it). Abby seems to already have the data transfer set up using USB, and John had mentioned that he was using UART for another subsystem on the microcontroller so hopefully we can proto before finalizing the PCB.

All that to say, it fell on me to research and layout the USB à UART translator. My initial thought with this was to go with the ECE362 module (FT232R chipset). After some research, it appears this chipset is limited to 3 MBaud which I misinterpreted to be a significantly slower data rate than it actually is. Normally I wouldn’t include an error in a report, but my error cost me nearly an hour trying to find a faster chipset before re-doing my math and realizing that 3MBaud would have plenty of throughput for the application. See this article for learning process on baud rates.

After making this decision, I then found the footprint and hooked it up into the current iteration of the PCB schematic. Note that I’ve also added the JTAG programmer connection in the bottom right.

 

Microcontroller to 7 Segment Display (3 hours)

For the heads-up display (HUD) we will be using the existing but broken hardware that came with the table. Having done some low-cost displays like this before, it fell on me to understand how to interface with the existing PCB via the 2 8 pin connectors. Took about an hour in the lab with the benchtop power supplies (and a good amount of research on the operating voltages of the 7 segment displays to make sure I didn’t burn them out) to reverse engineer my way into lighting the displays. Once this was done, I began making a prototype connection to run this off of the microcontroller.

In short, the two 8 pin connectors are the display selector and the segment selectors respectively. Meaning that if you pull the display selector low (as the displays are common cathode) and n many of the segment selectors high, you can turn on any n of the 7 segments in a single one of the displays. For the control of this, we will therefore be exploiting persistence of vision to briefly turn each display on in sequence in the required pattern for score display.

To accomplish this, I have drawn up a simple SPI/shift register interface for the microcontroller.

Note that there will be an inverter after the display selector, but I need to proto physically before designing in schematic.

Other work (6 hours)

Once again did a bunch of work to sort physical issues out on the table. Filling the original goal hole so that we can move it back was almost the entire lab section this week. Once again, my forearms burn from the physical labor involved in cutting the new surface to size.

I was also responsible for the electrical overview, and spent roughly 3 hours on that over the course of the week.

Week 2 Progress Report

Date: 8/28-9/3/2023

Total Hours: 15

Description of Design Efforts:

This week’s primary task was to narrow the exact scope of the project to write complete PSDRs, continue investigating the mechanical systems to find the method of XY control, and start learning KiCad. The narrowing of scope was a team effort and can be summarized by reading the A1 documentation on the group website.

Continuing investigation of mechanical systems

While the motors and motor power systems were set last week, the actual arrangement of pulleys/gantry needed to be finalized this week. Ultimately, we settled on the CoreXY system for rapid motion in a 2D space. This method allows both motors to be fixed in space (unlike a classical gantry system which requires one of the two motors to be moved). Hopefully, this should allow for much faster acceleration of the mallet on the surface of the table. Additionally, it is important to note with this system that teethed belts and pulleys will be used rather than cables and cable spools. This should increase the XY motion accuracy of the system.

Week 1 Progress Report

Date: 8/21-8/27/2023

Total Hours: 10

Description of Design Efforts:

First week of the semester was fairly uneventful in terms of actual design work. Primary task focused on setting up the course-required website and design journals.

Research of mechanical systems

I took charge of mechanical systems research to find the required torque, speed, and accuracy we need out of our stepper motors along with the pulley/gantry arrangement and motor power/control systems. We are hoping for a rough max acceleration of 20 m/s^2 and a max speed of 27.2 m/s (60mph) while also maintaining a step accuracy down to .5 inches (I.E., one step of the motor will produce .5 inches or less of linear motion). For this system, I believe that I have settled on a NEMA 23 Bipolar Lathe Router, a DM542T CNC Stepper Driver, and am still doing research into the exact pulley system to be implemented.

Website Setup

I also took charge of the setup of the group website. From the provided template, I added all team-member information and set up the github repositories (one for website data, one for source code) in a format that would be easily accessed by my team members to update their own progress reports.
